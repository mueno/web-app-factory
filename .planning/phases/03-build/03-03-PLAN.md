**Change 2: Extend `_run_gate_checks()` to dispatch by gate type.**

The current implementation ONLY checks `required_files` and `required_output_markers` conditions — it does NOT dispatch to gate executor functions for gate types like `build` and `static_analysis`. The YAML contract defines these gate types for phases 2a and 2b, so they would silently pass without actual checking. This is a critical gap.

Refactor `_run_gate_checks()` to:
1. Iterate over each gate in `contract_phase.get("gates", [])`
2. Read `gate.get("type")` to determine the gate type
3. Dispatch based on gate type:
   - `"artifact"`: Check `conditions.required_files` exist (current behavior for required_files)
   - `"tool_invocation"`: Check `conditions.required_output_markers` in docs/pipeline/ (current behavior for markers)
   - `"build"`: Import and call `run_build_gate(project_dir, phase_id=contract_phase["id"])`. If `GateResult.passed` is False, extend issues with `GateResult.issues`
   - `"static_analysis"`: Import and call `run_static_analysis_gate(project_dir, phase_id=contract_phase["id"])`. If `GateResult.passed` is False, extend issues with `GateResult.issues`
   - Unknown type: Add issue `f"Unknown gate type: {gate_type}"` (fail-closed per gate_policy)

Import the gate functions at the top of the file:

```python
from tools.gates.static_analysis_gate import run_static_analysis_gate
```
