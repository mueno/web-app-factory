    - Two-phase endpoint (challenge generation + verification):
      - Phase 1 (GET or POST without attestation): Generate registration options via `generateRegistrationOptions()` from `@simplewebauthn/server`. Options include rpName (from env or app name), rpID (from request host), challenge (auto-generated), user info (from request body: email/name). Store challenge in a server-side session or short-lived cookie.
      - Phase 2 (POST with attestation): Verify registration response via `verifyRegistrationResponse()`. On success, store the credential (credentialID, publicKey, counter) in Supabase `passkey_credentials` table using the service_role client. Return success.
    - Use `createServerClient` with service_role key (server-side only, admin operations)
    - Store credentials in a `passkey_credentials` table (credentialID as primary key, user_id FK, public_key, counter, transports, created_at)

    **auth/passkey-auth-api.ts.tmpl**: POST route handler for WebAuthn authentication ceremony.
    - File location: `src/app/api/auth/passkey/authenticate/route.ts`
    - Two-phase endpoint:
      - Phase 1 (POST with email/userId): Look up user's registered credentials from `passkey_credentials` table. Generate authentication options via `generateAuthenticationOptions()` with `allowCredentials` set to the user's registered credential IDs. Store challenge.
      - Phase 2 (POST with assertion): Verify authentication response via `verifyAuthenticationResponse()` against stored credential. On success, update counter in DB. Then call the passkey hooks module to create a Supabase session.
    - Return the Supabase session tokens (or set cookies) so the client is authenticated

    **auth/passkey-hooks.ts.tmpl**: Custom Supabase JWT session creation after passkey verification.
    - File location: `src/lib/auth/passkey-hooks.ts`
    - Export `createPasskeySession(userId: string)` function
    - Use Supabase Admin API (`supabase.auth.admin.generateLink({ type: 'magiclink', email })`) to create a one-time login link, then exchange it via `exchangeCodeForSession`. This is the recommended pattern for bridging external auth (WebAuthn) into Supabase sessions without storing passwords.
    - Alternative approach (document both, implement the more reliable one): Use `supabase.auth.admin.createUser()` to ensure the user exists, then `supabase.auth.signInWithPassword()` with a server-generated one-time token stored in the user's metadata.
    - The service_role key is required and is ONLY used server-side (never exposed to client)
    - Include `// SECURITY: This file must only run server-side` comment

    **auth/passkey-client.tsx.tmpl**: Client-side passkey UI component.

    - Two buttons: "Register Passkey" and "Sign in with Passkey"
    - Registration flow: POST to `/api/auth/passkey/register` to get options, call `startRegistration(options)` from `@simplewebauthn/browser`, POST attestation back to `/api/auth/passkey/register`
    - Authentication flow: POST to `/api/auth/passkey/authenticate` to get options, call `startAuthentication(options)` from `@simplewebauthn/browser`, POST assertion back to `/api/auth/passkey/authenticate`
    - On successful auth, redirect to `returnTo` path (from props or searchParams)
    - Style with Tailwind — passkey button should be visually prominent (primary action per user decision: passkey is the main auth method)
    - Handle errors gracefully: show user-friendly error messages for "NotAllowedError" (user cancelled), "SecurityError" (domain mismatch), etc.
    - Include fingerprint/key icon with the passkey buttons (emoji or SVG)

    CRITICAL DESIGN NOTES:
    - Supabase does NOT natively support WebAuthn — this is a bridge pattern using custom API routes + @simplewebauthn
    - Credential storage goes in a Supabase table, not Supabase Auth's built-in credential storage
    - Session management bridges to Supabase Auth via admin API (generateLink or similar)
    - The `passkey_credentials` table needs RLS: users can only read/write their own credentials
    - All server templates use `getUser()` (not `getSession()`) where session validation is needed
