From tools/phase_executors/base.py:
```python
class PhaseExecutor(ABC):
    @property
    @abstractmethod
    def phase_id(self) -> str: ...
    @property
    @abstractmethod
    def sub_steps(self) -> list: ...
    @abstractmethod
    def execute(self, ctx: PhaseContext) -> PhaseResult: ...

@dataclass(frozen=True)
class PhaseContext:
    run_id: str; phase_id: str; project_dir: Path; idea: str; app_name: str
    extra: dict = field(default_factory=dict)

@dataclass
class PhaseResult:
    phase_id: str; success: bool; artifacts: list; sub_steps: list
    error: str | None = None; resume_point: str | None = None

@dataclass
class SubStepResult:
    sub_step_id: str; success: bool; artifacts: list
    error: str | None = None; notes: str = ""
```

    ```
    class Phase1bSpecExecutor(PhaseExecutor):
        phase_id = "1b"
        sub_steps = ["load_context", "write_prd", "derive_screen_spec", "cross_validate", "self_assess"]

        def execute(self, ctx: PhaseContext) -> PhaseResult:
            1. Load Phase 1a output as context:
               - Read idea-validation.md (if exists) for competitor/market context
               - Read tech-feasibility-memo.json (if exists) for tech constraints
            2. Load quality criteria for "1b" from YAML contract
            3. Build phase-specific system prompt from SPEC_AGENT.system_prompt + quality criteria
            4. Construct user prompt incorporating:
               - ctx.idea and ctx.app_name
               - Phase 1a context: embed idea-validation.md content and tech-feasibility-memo.json content directly into the prompt (not just reference paths)
               - Instruction to write prd.md FIRST, then derive screen-spec.json from it
               - Instruction for MoSCoW labels on every requirement
               - Instruction for component inventory with parent-child hierarchy
               - Instruction for exact component name matching between PRD and screen-spec.json
            5. Call run_spec_agent() with the prompts
            6. Verify deliverables exist
            7. Cross-validate:
               - Parse screen-spec.json
               - Extract component names
               - Check each appears in prd.md component inventory
               - If mismatch found: return PhaseResult(success=False)
            8. Generate quality self-assessment (CONT-04)
            9. Return PhaseResult with artifacts list

        Helper methods:
        - _build_user_prompt(ctx, phase_1a_context: dict) -> str
        - _load_phase_1a_context(project_dir: Path) -> dict
        - _validate_deliverables(project_dir: Path) -> tuple[bool, list[str]]
        - _cross_validate_components(project_dir: Path) -> tuple[bool, list[str]]
    ```

    - Reads prd.md, extracts component names from "## Component Inventory" section
    - Reads screen-spec.json, extracts all component names from screens[].components
    - Compares: every component in screen-spec.json must appear in PRD inventory
    - Returns (passed, mismatches_list)
