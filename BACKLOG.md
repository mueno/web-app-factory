### BL-001: Phase 2b の WBS 分解（増分実装）

**優先度:** P1（次に着手）
**起票:** 2026-03-23
**背景:**

Phase 2b（Build）は現在、build エージェントが全ページ・全コンポーネントを一括で生成する。
この方式には以下の問題がある:

1. **失敗時のリカバリコストが高い**: 2b が途中で失敗すると、全実装がゼロからやり直しになる
2. **所要時間が長い**: 10〜20分かかり、途中経過が見えない
3. **デバッグが困難**: どのコンポーネントで問題が起きたか切り分けにくい

**検討すべきアプローチ:**

- **案A: コンポーネント単位の分割実行**
  - shared components → pages → layouts の順に個別実行
  - 各ステップで `npm run build` を挟み、ビルド可能性を保証
  - 失敗時は失敗したコンポーネントのみ再生成

- **案B: screen-spec.json ベースのページ単位実行**
  - screen-spec.json の各画面をサブタスクとして分解
  - 1画面ずつ生成 → ビルド検証 → 次の画面へ
  - checkpoint を画面単位で保存し、失敗時はそこから再開

- **案C: レイヤー別の段階的実装**
  - Step 1: 型定義・データモデル
  - Step 2: 共通コンポーネント
  - Step 3: ページ実装
  - Step 4: 結合・ルーティング
  - 各ステップ完了時にビルドゲートを通す

**影響範囲:**
- `tools/phase_executors/phase_2b_executor.py`
- `contracts/pipeline-contract.web.v1.yaml`（Phase 2b のサブフェーズ定義追加）
- `tools/pipeline_state.py`（サブフェーズ単位の checkpoint 対応）

**成功基準:**
- Phase 2b 失敗時、最後に成功したサブステップから再開できる
- 各サブステップ完了時にビルドが通ることが検証される
- 総所要時間が一括実行と比較して1.5倍以内に収まる

### BL-002: E2E フロー検証ゲート（Playwright ベース）

**優先度:** P1（次に着手）
**起票:** 2026-03-23
**背景:**

Five Whys 分析（推し活シミュレーター FLOW-01 バグ）により、パイプラインに
ランタイムレベルのフロー検証が一切ないことが判明。現行ゲートはすべて
コンパイル時・静的解析レベル。フォーム送信→結果表示のような複数ページに
またがるユーザーフローは、デプロイ後にユーザーが触るまで検出できない。

**対策案:**
- Phase 3（Ship）のゲートに Playwright E2E テストを追加
- PRD / screen-spec から主要ユーザーフローを自動抽出
- フォーム入力 → 送信 → 結果ページ表示を自動検証
- 検証対象: フォーム送信、ページ遷移、searchParams の受け渡し

**影響範囲:**
- `tools/gates/` に新規ゲート追加
- `contracts/pipeline-contract.web.v1.yaml` Phase 3 ゲート定義
- `tools/phase_executors/phase_3_executor.py` ゲート呼び出し追加

### BL-003: Phase 1b で画面間データフロー契約を仕様化（済: 部分対応）

**優先度:** P2（設計先行必要）
**起票:** 2026-03-23
**背景:**

screen-spec.json は画面のレイアウト・コンポーネント・状態を定義するが、
画面間のデータフロー（URL パラメータ、フォーム送信先と受信パラメータ名）を
定義する仕組みがない。build エージェントが各画面を独立に生成するため、
画面間のパラメータ名が不一致になる。

**部分対応（済）:**
- FLOW-01 静的解析ゲートで事後検出は可能になった
- build エージェントプロンプトに共有インターフェース指示を追加済み

**残課題:**
- screen-spec.json スキーマに `inputParameters` / `outputParameters` を追加
- Phase 1b の spec エージェントプロンプトにデータフロー定義の指示を追加
- build エージェントが spec のパラメータ定義を参照して実装するフロー
